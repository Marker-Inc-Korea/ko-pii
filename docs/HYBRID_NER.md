# 하이브리드 PII 검출 — 룰(ko-pii) + ML NER

ko-pii는 **결정적 ID(주민·카드·사업자·전화·이메일 등)를 체크섬으로 ~F1 1.0** 잡지만,
**퍼지 카테고리(이름·주소·직책·학력 등)는 약합니다**(KDPII PERSON 0.135 등). 이 약점을
**한국어 토큰분류 NER**로 보완하는 하이브리드를 실험·평가했습니다.

> 설계 원칙: **ko-pii가 결정적 ID 담당(체크섬, ML이 못 이김) · ML이 퍼지 담당.** 교체가 아니라 보완.

## 실험 설정

- **학습 데이터**: KDPII `train`(대화체 40,109문서) + 생성 행정 평가셋의 학습분 1,398문서.
  결정적 ID는 제외하고 **퍼지 20종**(PERSON·ADDRESS·POSITION·EDUCATION·MAJOR·NATIONALITY·
  AGE·DT_BIRTH·HEIGHT·WEIGHT + KDPII 고유 NICKNAME·WORKPLACE 등)만 토큰분류.
- **평가(held-out)**: KDPII `test`(4,891) + 생성 검증셋 540(행정). 동일 매처(`match_forms_overlap`, PML=3).
- **누수 방지**: KDPII **원본 train/test 파티션**을 그대로 사용(재분할 X), 검증 540은 학습에서 제외.
  (KDPII 원본엔 짧은 공통 발화 ~71개가 train/test 중복이나 *PII 보유는 1개* — 사실상 무누수.
  `ko_pii.eval.dataset_integrity.assert_no_text_leakage` 로 빌드타임 검증.)
- **재현**: 데이터는 레포 `data/`(KDPII + 생성셋). 정확한 레시피·명령은 아래 [재현(레시피)](#재현-레시피) 절.

## 베이스 모델 + 하이퍼파라미터 ablation — 퍼지 NER 단독 (seqeval)

| 베이스 | lr·epoch | KDPII fuzzy | 생성 fuzzy | 추론속도 | 판정 |
|---|---|---|---|---|---|
| **klue/roberta-large** (한국어 336M dense) | **2e-5·3ep** | **0.937** | **0.862** | **7.4ms** | ✅ **채택 (포화점)** |
| klue/roberta-large | 2e-5·6ep | 0.935 | 0.844 | 7.4ms | 에폭↑ 효과 없음 |
| klue/roberta-large | 3e-5·6ep | 0.928 | 0.859 | 7.4ms | 고 lr 오히려 ↓ |
| openai/privacy-filter (1.4B MoE) | 1e-4·5ep | 0.633 | 0.437 | 77.5ms | ❌ 학습은 되나 열위 |
| openai/privacy-filter | 2e-5·3ep | 0.138 | 0.209 | 77.5ms | (과소학습 — 아래 ② 참조) |
| klue/roberta-large **무학습(base)** | — | 0.001 | 0.000 | 7.4ms | 헤드 랜덤 — 실측 0 |

**① klue는 3 epoch 에서 포화.** 6 epoch·고 lr 모두 3ep(KDPII 0.937 / 생성 0.862)을 못 넘었다 — 추가 학습은
한국어 dense BERT 의 천장에 부딪혀 오히려 과적합 쪽(6ep 생성 0.844, 6ep·3e-5 KDPII 0.928).
**2e-5·3ep 이 정확도·비용 동시 최적 → 프로덕션 레시피.**

**② openai/privacy-filter 의 0.138 은 "구조적 부적합"이 아니라 under-training 이었다(정정).**
초기 lr 2e-5 에선 0.138 에 머물렀으나 **lr 1e-4 로 올리니 KDPII 0.633·생성 0.437 로 ~4.6배 상승** —
모델은 분명히 학습된다. 다만 **1.4B Mixture-of-Experts(층당 128 experts) + 어휘 200K + 131K 롱컨텍스트**
구조라 용량이 다국어 expert 전반에 분산돼, 한국어 특화 **336M dense klue 의 한 forward**를 못 이긴다
(KDPII 0.633 ≪ 0.937). 게다가 한 forward 가 본질적으로 무거워 **추론 10배 느림(77.5 vs 7.4ms/doc)**.
→ *정확도로도 속도로도* 한국어 컴팩트 BERT 가 정답. (lr 1e-4 검증 전의 "부적합" 판정은 과소학습 산물이었음.)

## 전후 비교 — 룰단독 vs 하이브리드 (전체 카테고리, match_forms_overlap)

프로덕션 모델(**klue 2e-5·3ep**, 단독 퍼지 0.937) 기준:

| 평가셋 | ko-pii **룰단독** | **ko-pii + klue 하이브리드** | Δ |
|---|---|---|---|
| KDPII test (대화체) | 0.659 | **0.848** | **+0.189** |
| 생성 540 (행정체) | 0.790 | **0.926** | **+0.136** |

**하이브리드(ko-pii 체크섬ID + klue 퍼지)가 KDPII에서 0.848 → 자체호스팅 Gemma-4-31B(0.796)마저 능가**
(게다가 결정적·온프레미스·체크섬 검증), 생성셋 0.926 → Gemma-31B(0.964) 근접.

## 핵심 매트릭스 — 단독 × 하이브리드 × base/tuned (풀 파이프라인)

| 구성 | KDPII | 생성540 | vs 룰 |
|---|---|---|---|
| ko-pii 룰단독 (기준) | 0.659 | 0.790 | — |
| openai/PF **base** 단독 (zero-shot 퍼지) | 0.264 | 0.451 | — |
| ko-pii + openai/PF **base** (하이브리드) | 0.426 | 0.647 | **−0.23 / −0.14** ❌ |
| ko-pii + openai/PF **tuned**(lr1e-4) (하이브리드) | **0.748** | **0.732** | **+0.09 / −0.06** △ |
| **ko-pii + klue tuned** (하이브리드) | **0.848** | **0.926** | **+0.19 / +0.14** ✅ |

- **base(zero-shot) openai/PF**: 하이브리드가 룰을 **양쪽 다 떨어뜨림** — 나쁜 ML 이 ko-pii 의 *좋은* 퍼지
  검출(예: 생성 ADDRESS 0.966)을 *나쁜* 것으로 교체하기 때문.
- **tuned openai/PF**: 대화체(KDPII)는 룰을 **상회(+0.09)** 하나 행정체(생성)는 **−0.06** 으로 혼조. 퍼지 recall 이
  낮아(생성 fuzzy 0.437) ko-pii 의 강한 룰 ADDRESS 를 교체하면 손해 — **net marginal + 추론 10배 느림**.
- **tuned klue**: 양쪽 모두 큰 폭 개선 — 유일하게 *일관되게* 이득.

> **정정 노트.** 이전 버전은 openai/PF tuned 하이브리드를 "룰 미만 확실"로 *미측정 추정*했으나, 실측 결과
> KDPII 0.748 로 룰(0.659)을 **상회**한다(행정체만 −0.06). 추정이 틀렸고 본 표는 실측치로 교체했다.

## 확장 평가 — 학습 라벨 범위 × 채점 시각

"퍼지만 학습"이라는 설계 가정과 "하이브리드"라는 결합 방식 자체를 ablation 했다.
**학습 범위** = 퍼지 20종(설계 기본) vs **전체 36종**(결정적 ID 16종 포함), **채점** = 4모드
(룰단독 / ML단독 / 하이브리드 / 유니온). 원시 코드·로그·결과: [`experiments/ner/`](../experiments/ner/).

**단독 seqeval — 학습 범위 효과:**

| 모델 | KDPII | 생성540 |
|---|---|---|
| klue 무학습 base | 0.001 | 0.000 |
| klue 퍼지 20종 학습 | 0.937 | 0.862 |
| **klue 전체 36종 학습** | **0.950** | **0.882** |
| └ 그중 퍼지 서브셋만 채점 | 0.936 | 0.865 |
| gemma-4-E2B 전체 36종 (lr 5e-5) | 0.353 | 0.701 |
| gemma-4-E2B 전체 36종 (lr 1e-4) | 0.351 | 0.673 |

→ **결정적 ID 라벨을 추가해도 퍼지 성능 무손실**(0.936 ≈ 0.937), 비용은 학습 15→29분뿐.

**디코더 LLM 파인튜닝 (gemma-4-E2B, 4.6B 텍스트타워).** 매트릭스에 비어 있던 마지막 칸 —
"신형 소형 LLM 을 *직접 파인튜닝*하면?" — 을 실측으로 채웠다. transformers 에 토큰분류 클래스가
없어 커스텀 헤드(bf16 풀 파인튜닝)로 동일 데이터·동일 라벨(전체 36종) 학습:

- **두 lr(5e-5/1e-4) 모두 valid F1 ~0.33 에서 조기 정체** — openai/PF 때와 달리 lr 5배에도
  천장이 같아 **과소학습이 아니라 구조적 한계**로 확정 (klue 는 에폭 1에 0.92).
- 행정체(0.70)가 대화체(0.35)보다 훨씬 좋다 — 정형 ID 패턴은 배우지만 **대화체 퍼지 경계
  판정에서 무너지는** 패턴. 단방향(causal) 어텐션이 우측 문맥을 못 보는 디코더의 약점 및
  262K 다국어 어휘의 용량 분산과 일치하는 결과.
- 비용도 열위: 학습 klue 의 ~4~8배(72~112분), 파라미터 14배.

→ openai/PF(1.4B MoE, 0.633)·gemma-4-E2B(4.6B 디코더, 0.353) 두 실측으로, **한국어 양방향
인코더의 우위는 "더 새롭고 큰 모델"로 뒤집히는 문제가 아님**이 반복 확인됐다.

**채점 매트릭스 — 전체 카테고리, 프로토콜 v2** (룰단독 기준선 0.659 / 0.790):

| 구성 (KDPII / 생성540) | ML단독 | 하이브리드(룰ID+ML퍼지) | 유니온(룰∪ML) |
|---|---|---|---|
| klue **base** (무학습) | 0.002 / 0.005 | 0.059 / 0.110 | 0.108 / 0.195 |
| klue 퍼지학습 | 0.685 / 0.661 | **0.904 / 0.927** | 0.799 / 0.863 |
| klue **전체학습** | **0.949 / 0.943** | 0.898 / 0.924 | 0.847 / 0.875 |

1. **전체학습 ML단독이 전 구성 최고** (KDPII 0.949·생성 0.943) — 단일 모델 시나리오가
   인도메인 벤치마크에선 하이브리드까지 능가한다. **단 캐비엇**: 생성540의 ID 패턴은 학습분과
   같은 생성기 출신(인분포)이고, ML은 **체크섬 검증이 불가**해 무효 번호를 구분 못 한다 —
   결정성·OOD 견고성이 필요한 배포에선 룰의 가치가 별개로 남는다.
2. **퍼지학습 모델은 하이브리드가 최적**(0.904/0.927) — 역할 분담 구조가 유효.
3. **유니온은 항상 하이브리드 이하** — 양쪽 FP 가 합산된다. 단 recall 은 모드 중 최대
   (전체학습 유니온: KDPII 0.937 / 생성 0.965)라 *놓치면 안 되는* 보수적 마스킹 용도로는 의미 있음.
4. **무학습 base 는 어느 모드에서든 룰을 파괴**(하이브리드 0.659→0.059) — openai/PF base(−0.23)와
   함께, 하이브리드의 하방이 ML 품질에 뚫려 있음을 재확인.

> **채점 프로토콜 노트 (v1→v2).** 위 "전후 비교"·"핵심 매트릭스"(v1)는 ML 예측의 gold 라벨공간
> 보정 없이 채점했다 — KDPII gold 엔 NATIONALITY 라벨이 없는데(LCP_COUNTRY→ADDRESS) ML 의
> NATIONALITY 예측을 FP 로 처벌. 본 절(v2)은 ML 예측을 **데이터셋 gold 라벨공간으로 한정**하고
> span 을 정규화(strip, PERSON<3자 제외)했다. 같은 klue 퍼지학습 하이브리드가 v1 0.848 → v2 0.904
> (KDPII)로 오르고 생성540 은 0.926→0.927 로 불변 — 차이의 원인이 라벨공간 보정임을 검증.

### 검증 시각 (3안: ML 검출 + 룰 체크섬 검증) — 그리고 벤치마크 gold 의 한계

"검출은 ML, ID 진위 검증은 룰" 조합을 측정했다 — 전체학습 ML 검출에 룰 체크섬(CARD =
Luhn+BIN 화이트리스트, 사업자·법인 = 검증자리)을 후처리 필터로 적용. **RRN 은 검증에서 제외**:
2020-10-05 이후 발급분은 뒷자리 무작위화로 체크섬이 진위 판정이 못 되며, ko-pii 도 RRN
체크섬을 신뢰도 신호로만 쓴다(거부 안 함).

| klue 전체학습 (KDPII / 생성540) | ML단독 | ML+룰검증(3안) |
|---|---|---|
| F1 | 0.949 / 0.943 | 0.919 (−0.030) / 0.929 (−0.014) |

벤치마크에선 3안이 손해인데, **그 이유가 이 평가 전체의 한계를 드러낸다 — gold ID 대부분이
"형식만 맞는 가짜 번호"다**: KDPII CARD 는 9%(7/78)만 Luhn+BIN 유효, 생성540 은 RRN 15%·
사업자 13%·법인 19%만 체크섬 유효. 검증 필터가 드롭한 것은 거의 전부 "가짜지만 gold"
(KDPII CARD: TP 손실 71 vs FP 제거 0). 반면 gold 가 95% 유효한 생성540 CARD 에선 3안이
**FP 7개 제거 vs TP 2개 손실로 이득** — gold 가 진짜 번호에 가까울수록 검증이 유리해지는 구조.

같은 아티팩트가 **룰 기준선도 깎고 있었다**: 룰의 생성540 사업자 recall 0.129 는 gold 체크섬
유효율 13%와 일치 — 룰은 실존 불가능한 번호를 거부(설계)했는데 벤치마크는 이를 FN 으로 처벌.
즉 **ML단독 우위(0.949/0.943)의 일부는 생성기의 가짜 번호 분포를 학습한 합성 아티팩트**이며,
실데이터(유효 체크섬)에선 룰·하이브리드·3안의 상대 위치가 올라갈 것이다. 도메인 일반화 평가
(모델 공개 선행 조건)가 필요한 또 하나의 이유.

**체크섬 프로브 (정성, [`experiments/ner/`](../experiments/ner/) 수록):** 무효 카드(BIN 1xxx)는
룰이 거부·ML 은 검출(진위 검증 불가) ↔ RRN 공백 변형(`850101 - 1234566`)은 룰이 놓치고 ML 이
검출(포맷 일반화) — 두 검출기의 실패 모드가 상보적이다. 같은 디렉토리의 마스킹 시연
(`mask-demo.log`)은 룰/ML 검출 span 에 동일한 tokenize·redact 후처리가 꽂힘을 보여준다
(마스킹은 검출기와 독립적인 후처리 계층).

## 외부 검증 (OOD) — 아티팩트를 제거하면 순위가 뒤집힌다

위 인분포 평가의 한계 3가지(가짜 gold ID·동일 분포·자기충족 anchor)를 고친 외부 검증셋
2종을 구축해 전 구성을 재평가했다.

- **Set A — OOD 주입 v2** (`data/corpus/external_inject_v2.jsonl`, 400문서·3,320 spans):
  base = AI Hub **공공 민원 상담**(기존 주입 코퍼스·학습데이터 미사용 도메인, 사전 redaction),
  **ID 전부 체크섬 유효**(`compute_check_digit` 역생성 — RRN 446·사업자 164·법인 164·카드 140,
  공백 변형 RRN 150 포함), **anchor 8종 신규**(기존 10종 미재사용).
  빌더: `data/build_external_eval_v2.py` (seed 고정, 재현 가능).
- **Set B — KLUE-NER dev** (5,000문장, 뉴스, **인간 라벨**): 우리가 만들지 않은 gold.
  기존 관행(한글 풀네임만, partial overlap)대로 PERSON 만 평가 — 단 룰 사전이 과거
  KLUE-NER FP 분석으로 보강된 이력이 있고 PS 라벨이 공인·가상인물을 포함해 보조 지표.

**Set A (전체 카테고리, match_forms_overlap):**

| 구성 | F1 | P | R | ID recall (RRN/CARD/사업자/법인) |
|---|---|---|---|---|
| 룰단독 | 0.892 | 0.926 | 0.861 | **1.00 / 1.00 / 1.00 / 1.00** |
| ML단독 (퍼지학습) | 0.611 | 0.978 | 0.444 | 0 (ID 미학습) |
| ML단독 (전체학습) | 0.929 | 0.958 | 0.902 | 0.94 / 0.93 / 0.86 / 0.85 |
| **하이브리드 (룰+퍼지ML)** | **0.968** | **0.990** | 0.947 | 1.00 × 4 |
| 하이브리드 (룰+전체ML) | 0.966 | 0.990 | 0.943 | 1.00 × 4 |
| 3안 (전체ML+룰검증) | 0.927 | 0.963 | 0.893 | 0.94 / 0.93 / 0.85 / 0.68¹ |

¹ 검증이 ML 의 경계 부정확 span(겹침 매처는 TP 로 인정하나 숫자가 정확히 13자리가 아님)을
드롭 — 검증 시각은 "겹치면 OK"가 아니라 정확한 값 추출을 요구한다는 부수 발견.

**Set B (KLUE-NER PS, 풀네임 관행):** 룰 0.382 / ML 퍼지학습 **0.834** / ML 전체학습 **0.840**
(P 0.95~0.97) — ML 퍼지가 뉴스 도메인 인간 라벨에서도 유지됨. KDPII 암기가 아니라 일반화.

**결론 — 인분포 우위는 아티팩트였다.**

1. **룰 0.659/0.790 → 0.892 복원**: gold 가 진짜(체크섬 유효) 번호가 되자 룰 ID recall 이
   전부 1.00. 인분포 벤치마크가 룰을 체계적으로 과소평가했음이 확증.
2. **하이브리드가 외부 최강(0.968)**: 인분포 챔피언 ML단독(0.949)은 외부에서 0.929 로
   하이브리드에 역전당함. ML 의 ID recall(0.85~0.94)이 룰 체크섬(1.00)을 못 따라가는 게 원인.
3. **ML 퍼지의 가치는 진짜**: 외부에서도 PERSON 등 퍼지는 ML 이 압도(Set B 0.84 vs 0.38) —
   하이브리드 구조(룰=ID, ML=퍼지)가 양쪽의 *진짜* 강점만 취한 구성임이 외부 데이터로 검증됨.

→ **프로덕션 확정: klue 기반 하이브리드** (모델은 전체학습판 권장 — 하이브리드 성능 동급
(0.966≈0.968)이면서 ML단독 폴백(0.929)까지 가능). ML단독은 인분포 + 마스킹 한정 옵션으로 강등.

## 결론 (중요)

- **좋은 한국어 모델을 튜닝**해야 하이브리드가 *일관되게* 효과(klue: 룰 대비 +0.14~0.19).
- **base(zero-shot) 모델은 역효과** — openai/PF base 하이브리드는 룰보다 **−0.23/−0.14**.
- **부적합으로 보였던 openai/PF 도 제대로 학습하면(lr 1e-4) 대화체에선 이득(+0.09)**, 단 행정체 손해(−0.06)·
  추론 10배 느림으로 **한국어 컴팩트 klue 에 종합 열위** → 불채택. (불채택 사유 = "학습 불가"가 아니라 "열위".)
- 즉 "ML 얹으면 좋아진다"가 아니라 **"제대로 튜닝된 한국어 NER이라야 *일관되게* 좋아진다"**.
- **결합 방식도 측정 대상이다**: 유니온(룰∪ML)은 하이브리드(역할 분담)에 항상 진다 — FP 합산.
  결합은 "다 합치기"가 아니라 **각자 잘하는 영역만 맡기기**가 정답.
- **전체 36종 학습 시 ML 단독이 인도메인 최고**(0.949/0.943)였으나, **외부 검증(OOD·체크섬 유효
  gold)에서 하이브리드(0.968)가 ML단독(0.929)을 역전** — 인분포 우위는 가짜 gold 아티팩트였음이
  확증됐다(위 외부 검증 절). **프로덕션 = klue 전체학습 + 하이브리드 확정.**
- **백본 교체로 풀릴 문제가 아니다** — PII 전용 MoE(openai/PF 0.633)도, 신형 소형 LLM 직접
  파인튜닝(gemma-4-E2B 0.353)도 klue(0.950)에 크게 못 미침. 현재 병목은 백본이 아니라
  **데이터(도메인 커버리지·gold 품질)**.

## 재현 (레시피)

별도 가중치·스크립트 배포 없이 레포만으로 재현 가능하다. 데이터는 레포 `data/` 를 그대로 쓴다.

- **학습 데이터**: `data/kdpii/{train,valid,test}.json` + `data/generated_eval_large.jsonl` 의 학습분
  (검증 540=`data/generated_eval.jsonl` 은 제외). 라벨은 **퍼지 20종만**(결정적 ID 제외).
- **모델·하이퍼파라미터(프로덕션)**: `klue/roberta-large` · **lr 2e-5 · 3 epoch** · max_len 256 · bs 32 ·
  warmup 0.1 · weight_decay 0.01 · `metric_for_best_model=f1`(epoch 평가). seqeval 로 퍼지 F1 측정.
- **하이브리드 평가**: ko-pii `detect_all` 의 결정적 ID 결과는 그대로 두고 **퍼지만** NER 예측으로 교체,
  동일 매처(`ko_pii.eval.kdpii.match_forms_overlap`, PML=3)로 룰단독 vs 하이브리드 전체 F1 비교.
- **openai/privacy-filter 재현 시 주의**: 반드시 **lr 1e-4 · bf16 · `trust_remote_code`** (lr 2e-5 는 과소학습 → 0.138).

> 토큰분류 학습 루프 자체는 HF `Trainer` + `AutoModelForTokenClassification` 표준 구성이며, 위 설정만
> 그대로 따르면 본 문서의 단독/하이브리드 수치가 재현된다(±0.003 무작위성).

실제 사용한 학습·평가 코드와 **전체 실행 로그(에폭별 loss·F1, 카테고리별 리포트)·원시 결과 JSON** 은
[`experiments/ner/`](../experiments/ner/) 에 아카이브돼 있다 — `--all-labels`(전체 36종 학습) 변형 포함.

라이선스 검토(KDPII·생성셋 재배포 가능 여부)와 도메인 일반화 평가 후 `[classifier]` extra로
