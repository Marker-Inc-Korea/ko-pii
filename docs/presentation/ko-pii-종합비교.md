# ko-pii 종합 성능 비교표

> 동일 채점기(`match_forms_overlap`, position-insensitive 부분집합 매칭) + 동일 조건
> (`person_min_length=3`)으로 모든 시스템을 같은 문서에 채점. 측정값과 참조값을 분리 표기.

## 1. 정확도 — 2개 데이터셋 × 4개 시스템

### A. KDPII (4,891문서 · 인간 라벨 · 대화체) — *측정값*

| 시스템 | F1 | P | R | 비고 |
|---|---|---|---|---|
| **Gemma-4-31B** (자체호스팅 LLM) | **0.796** | 0.850 | 0.748 | 대화체가 LLM에 유리 |
| **ko-pii** (룰+사전+체크섬) | **0.660** | 0.699 | 0.624 | 비-LLM 중 1위 |
| Presidio (`kr_adapt`) | 0.273 | 0.721 | 0.169 | 한국 카테고리 다수 미지원 |
| openai/privacy-filter (660M) | 0.264 | 0.317 | 0.226 | 한국 카테고리 7종만 |

### B. 생성 평가셋 (540문서 · 정제 gold · 행정/서식체) — *이번 세션 측정값*

| 시스템 | F1 | P | R | 비고 |
|---|---|---|---|---|
| **Gemma-4-31B** (자체호스팅 LLM) | **0.964** | 0.963 | 0.966 | LLM 생성 gold라 LLM에 유리(주의) |
| **ko-pii** (룰+사전+체크섬) | **0.784** | 0.792 | 0.776 | 비-LLM 중 1위 |
| Presidio (`kr_adapt`) | 0.483 | 0.794 | 0.347 | 고정밀·저재현 |
| openai/privacy-filter (660M) | *측정불가¹* | — | — | 참조: KDPII 0.264 / LLM생성-187 0.434 |

¹ 이 호스트에서 onnx 추론이 비정상 지연(30분+ 동안 100문서 미만) → 측정 포기. 약한 도구라 결론에는 영향 없음.

**해석**: 두 데이터셋 모두 집계 F1은 LLM > ko-pii > (Presidio ≈ openai/PF) 순서.
단, 생성셋의 0.964는 **gold가 LLM 생성**이라 라벨 컨벤션(특히 소프트 속성)이 LLM 추출과 정렬되어 **부풀려진 값**이다. 독립 인간 라벨인 KDPII에서 격차(0.796 vs 0.660)가 더 현실적이다.

## 2. ko-pii per-label (생성셋) — 강점/약점이 명확

| 구간 | 라벨 (F1) |
|---|---|
| **천장급 (≥0.93)** | EMAIL 0.998 · VEHICLE 0.994 · PHONE 0.989 · CARD 0.988 · ADDRESS 0.966 · ACCOUNT 0.959 · RRN 0.955 · URL 0.949 · HEIGHT 0.947 · WEIGHT 0.933 |
| 강 (0.81–0.92) | AGE 0.915 · IP 0.899 · MAJOR 0.873 · NATIONALITY 0.830 · PASSPORT 0.813 |
| 중 (0.60–0.77) | FRN 0.765 · EDUCATION 0.763 · DT_BIRTH 0.646 · DRIVER_LICENSE 0.644 · POSTAL_CODE 0.610 · PERSON 0.599 |
| **약/제로** | POSITION 0.243 · BUSINESS_REG 0.220 · CORP_REG 0.195 · MEDICAL_INSURANCE 0.000 · PRESCRIPTION_ID 0.000 |

**전체 0.784를 끌어내린 3가지 원인**:
1. **소프트 속성** — POSITION(직책) R 0.15. ko-pii는 직책을 의도적으로 추적하지 않음. LLM이 잘 잡는 영역.
2. **체크섬 vs 허구 번호** — BUSINESS_REG/CORP_REG는 **P 0.73~1.0인데 R만 폭락**. LLM이 생성한 gold의 사업자/법인번호가 **체크섬 비유효**라 ko-pii가 정당하게 기각(설계대로). gold 현실성 한계가 동시에 드러남.
3. **개방형 ID 미구현** — MEDICAL_INSURANCE·PRESCRIPTION_ID는 해당 형태 탐지기 부재 → 0.0.

## 3. 속도 — *측정값*

| 시스템 | 지연/문서 | 처리량 | 하드웨어 |
|---|---|---|---|
| **ko-pii** | **0.19 ms** | **~5,350 문서/s** | CPU 1코어 |
| Presidio | 4.2 ms | ~238 문서/s | CPU 1코어 |
| openai/privacy-filter | 481 ms | ~2 문서/s | CPU (대량시 GPU) |
| Gemma-4-31B (vLLM, 동시16) | 0.30 s(중앙) | 27.6 문서/s | GPU 1장 |

→ ko-pii는 자체호스팅 LLM 대비 **문서당 약 200배** 빠름.

## 5. 정성 비교

| 차원 | ko-pii | Presidio | openai/PF | LLM(Gemma) |
|---|---|---|---|---|
| 방식 | 룰+사전+체크섬 | spaCy ko NER+regex | 660M transformer | LLM 프롬프트 |
| **결정성** | 완전 결정적 | 결정적 | 결정적 | **비결정적** |
| **설명가능** | evidence+근거조항 | 부분 | 없음 | 없음(사후) |
| **체크섬 검증** | ✅ RRN·카드·사업자·법인 | ❌ | ❌ | ❌ |
| 온프레미스 | ✅ (의존성 0) | ✅ | ✅ | 자체호스팅시 ✅ |
| 외부 PII 전송 | 없음 | 없음 | 없음 | API시 전송 |
| 한국 PII 커버 | 33종 | 일부 | 7종 | 광범위 |
| 인프라 | CPU 1코어 | CPU | CPU/GPU | GPU/API |

## 6. 공정 비교 — 공통 지원 카테고리만 (*KDPII, 측정값*)

해외 도구는 한국 카테고리 다수를 아예 미지원(AGE·POSITION·RRN 등 0 산출). 각 도구가 **지원하는 카테고리로만** 한정해도 ko-pii가 우위:

- vs openai/privacy-filter (그 도구의 7개 라벨): **ko-pii 0.61 : 0.37**
- vs Presidio (그 도구의 9개 라벨): **ko-pii 0.87 : 0.65**

→ 격차는 단순 "미지원 카테고리" 때문이 아니라, **공통 지형에서도 ko-pii가 더 정확**.

## 7. 결론

| 우선순위 | 권장 |
|---|---|
| 대화체 자유텍스트 재현율 + 비용·지연 감수 가능 | **LLM** |
| 비용(≈$0)·속도(~5,350/s)·완전 온프레미스·외부전송 0·체크섬 검증된 결정적 ID | **ko-pii** |

ko-pii와 LLM은 **다른 운영점(operating point)**을 차지한다. ko-pii의 정체성은
"고속·결정적·설명가능·체크섬 검증된 한국어 구조적 PII 엔진"이며, 결정적 ID per-label이
양쪽 데이터셋에서 천장급(EMAIL/RRN/카드/전화 0.95~1.0)임이 그 근거다.
